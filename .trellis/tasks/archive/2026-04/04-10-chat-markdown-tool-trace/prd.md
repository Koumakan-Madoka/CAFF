# PRD: Chat 主界面 Markdown 渲染与工具链路可视化

## 背景

当前主聊天室中的 agent 消息主要以纯文本方式展示：

- 代码块、列表、引用、链接等结构信息被压平，可读性一般
- 技术型回复在长文本场景下容易像缠在一起的电线，不利于快速扫描
- agent 底层通过 pi-mono CLI 执行的读文件、跑命令、写文件等工具动作没有清晰映射到最终消息
- 当 agent 回答异常、执行失败或卡住时，用户很难判断问题发生在“思考/组织语言”还是“工具执行链路”

目前最核心的改造机会有两个：
1. 让聊天室里的 agent 文本更像“正常内容”，支持安全的 Markdown 渲染
2. 让工具调用过程更像“可解释的调试视图”，支持按消息关联查看工具链路

## 目标

1. **提升消息可读性**：支持常见 Markdown 子集渲染，显著改善代码、列表、引用等内容阅读体验
2. **提升执行可观测性**：让用户能看到 agent 某条消息背后的工具调用摘要与关键步骤
3. **保持主聊天流清爽**：工具链路默认不抢主视觉，普通用户仍能以聊天视角浏览
4. **安全优先**：避免 Markdown 渲染带来 XSS 风险，避免工具详情泄露敏感信息

## 非目标

- 不把底层 CLI 全量日志默认直接平铺到主聊天流
- 不开放原始 HTML 任意渲染
- 不在本阶段重做整个聊天架构或所有房间/游戏面板
- 不在没有明确需求的前提下扩展为完整终端模拟器

## 交互方案

### 1. Agent 消息 Markdown 渲染
- 对 agent 发送到聊天室中的消息启用安全 Markdown 渲染
- 优先支持标题、段落、列表、引用、链接、行内代码、代码块、表格等常见子集
- 默认禁用原始 HTML，危险协议链接需拦截或清洗
- 长代码块、超宽内容要有明确的滚动/换行策略
- 渲染失败时优雅降级为纯文本，不吞消息

### 2. 工具调用链路卡片
- 每条 agent 消息下方可附带“工具链路”摘要卡片
- 默认展示精简摘要：工具数量、成功/失败状态、总耗时、是否存在重试
- 展开后按时间线或步骤流展示工具名、状态、耗时、输入/输出摘要
- 对失败步骤做视觉高亮，并尽量就近展示错误原因

### 3. 阅读模式与调试模式分层
- 主聊天区域优先保留自然语言可读性
- 工具链路细节默认折叠，避免对话刷屏
- 如有必要，支持“只看结果 / 查看过程”的轻量切换

### 4. 消息关联与数据安全
- 工具事件需要能稳定关联到具体 agent 消息、turn 或调用链分组
- 如果现有事件结构缺少可靠关联键（如 `callId` / `messageId` / `turnId`），需先补齐或定义映射规则
- 展示前对敏感参数、环境变量、认证头、token、绝对路径等做脱敏处理
- 处理流式事件、乱序到达、失败重试和超长输出截断

## 实现范围

### 主要文件（预估）
- `public/chat/message-timeline.js`
- `public/chat/conversation-pane.js`
- `public/styles.css`
- `server/api/agent-tools-controller.ts`
- `server/domain/runtime/agent-tool-bridge.ts`
- `lib/agent-chat-tools.ts`

### 可能涉及的配套文件
- `public/shared/*` 下与消息渲染或 UI 帮助相关的小型辅助模块
- `server/domain/conversation/turn/*` 中与消息/事件结构相关的少量适配

> 以上范围以后续 research 结果为准，但原则上应保持“主聊天 UI + 工具事件桥接”聚焦，不顺手扩散成大范围重构。

## 验收标准

- [ ] agent 的常见 Markdown 输出可稳定渲染，代码块、列表、引用、链接等结构清晰可读
- [ ] 恶意或危险 Markdown payload 不会执行脚本，原始 HTML 不会直接注入 DOM
- [ ] 每条 agent 消息都可以在 UI 中查看与之关联的工具调用摘要
- [ ] 工具链路默认折叠，不显著打断主聊天阅读流；展开后可查看关键步骤、状态和耗时
- [ ] 失败步骤可被快速识别，并展示简洁明确的失败原因或错误摘要
- [ ] 工具详情展示前完成基础脱敏，不直接泄露敏感 token、认证信息或不必要的绝对路径
- [ ] 流式更新、重试或乱序场景下，工具链路仍能保持稳定归组，不出现明显错位或重复
- [ ] Markdown 渲染失败时消息仍以纯文本方式展示，不影响对话可用性

## 风险与待确认问题

1. 当前聊天事件模型是否已具备将工具调用稳定映射到最终 agent 消息的字段？
2. 工具调用详情的“输入/输出展示粒度”应展示到什么程度，才能兼顾价值与降噪？
3. 是否需要在 UI 中提供全局调试开关，还是按消息局部展开即可满足大多数需求？
4. Markdown 方案使用现有依赖还是引入轻量新依赖，需要结合当前前端资源组织方式确认

## 实施建议

1. 先调研现有聊天消息结构、工具桥接事件和可关联字段，确认数据契约
2. 定义前端消息视图模型，明确“自然语言内容”和“工具链路事件”的分层关系
3. 落地安全 Markdown 渲染与样式策略，并保证失败时可降级
4. 实现工具链路摘要卡片与展开时间线，优先覆盖成功/失败/重试三类核心状态
5. 最后验证长消息、长链路、流式更新、敏感信息脱敏和性能表现
