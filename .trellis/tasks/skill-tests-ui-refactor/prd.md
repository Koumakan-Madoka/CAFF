# PRD: Skill Tests 前端界面重构

## 背景

当前 Skill Tests 界面承载了生成、创建、批量运行、详情诊断、运行历史、回归对比等高频操作，但主要信息被压在左右双栏里：

- 浏览器缩放到 `125% / 150%` 后，核心按钮和状态信息容易漂移到可视区域之外
- 用户需要在左侧列表和右侧详情之间来回找操作，视觉锚点不稳定
- 详情、历史、回归信息同屏堆叠，信息密度偏高，新用户不容易快速上手
- 当前交互可用，但还不够“稳”，尤其不适合长时间巡检或演示场景

## 目标

1. **提升可发现性**：核心操作在常见缩放比例下始终容易找到
2. **响应式优先**：窄宽度或高缩放时自动切换为单列优先布局
3. **降低信息密度**：将详情、历史、回归拆成更清晰的分区或页签
4. **保持功能完整**：不牺牲现有 Skill Tests 能力

## 非目标

- 不重做后端 Skill Test API
- 不新增新的测试判定逻辑
- 不对 `eval-cases` 其他非 Skill Tests 区域做大范围视觉重写

## 用户体验方案

### 1. 顶部固定操作条
- 将 `Skill` 选择、`Agent`、`模型`、`Prompt Version`、`生成测试用例`、`手动创建`、`批量运行` 集中到顶部 sticky 区域
- 顶部操作条优先只承载高频主操作，避免在 `125% / 150%` 缩放下自身拥挤或换行失控
- 页面滚动或缩放后，主要操作仍可见

### 2. 单列优先的主布局
- 默认按“上 → 下”流组织内容，而不是强依赖左右双栏
- 上半区放概览、筛选和用例列表
- 下半区放当前用例详情
- 宽屏下可增强为双列，但不能把关键操作藏到边缘

### 3. 详情区 Tab 化
- 将当前用例区域拆成 `概览`、`详情`、`运行历史`、`回归对比` 等页签
- 详情头部固定保留当前 case 的状态、最近结果摘要和主操作按钮，避免切换 Tab 后再次寻找入口
- 减少同屏信息量，增强定位失败原因的效率

### 4. 列表卡片就近操作
- 用例卡片直接提供 `运行`、`查看详情`、必要时的 `编辑` 入口
- 列表中补充短 `case id`、状态、最近运行时间，降低相似 prompt 的辨识成本
- 提供轻量搜索或过滤能力，至少支持按 `case id` 或 prompt 关键字快速定位

### 5. 更清晰的状态与空态
- 强化 `可运行 / 触发失败 / 待验证` 的视觉提示
- 为空列表、加载中、运行失败提供更明确的提示文案
- 空列表和失败态需要给出明确下一步操作，例如 `生成测试用例`、`手动创建`、`重试`

## 实现范围

### 主要文件
- `public/eval-cases.html`
- `public/skill-tests.js`

### 允许的小范围配套
- 与 Skill Tests 区块直接相关的样式和 DOM 结构调整
- 必要时补充极小的前端辅助数据映射，但避免引入新的后端依赖

## 验收标准

- [ ] 在桌面浏览器 `1366x768` 下的 `100% / 125% / 150%` 缩放场景中，核心操作按钮首屏可见，且无需横向寻找即可完成主要流程
- [ ] 页面在较窄宽度或高缩放下自动切换到单列优先布局，不依赖左右双栏；除局部表格区域外不出现页面级横向滚动
- [ ] 顶部固定操作条承载主要配置与全局操作，并在滚动时保持可见；高缩放下仍以高频主操作为主，不因拥挤而失去可用性
- [ ] 当前用例详情拆分为更清晰的页签或等效分区，显著降低同屏信息密度；切换页签后主操作按钮和状态摘要仍易于找到
- [ ] 用例列表提供短 `case id`、状态、最近运行时间和就近操作，并支持按 `case id` 或 prompt 关键字快速定位
- [ ] 新用户可在 3 步内完成“选 Skill → 运行单条 → 查看失败原因”的主流程
- [ ] 不破坏现有流程：生成、手动创建、单条运行、批量运行、查看详情、回归对比仍可用
- [ ] 关键状态在无数据、失败、加载中时均有清晰提示，并提供明确下一步操作

## 实施建议

1. 先重排页面骨架：顶部操作条 + 单列优先布局，并优先保证高频主操作在高缩放下稳定可见
2. 再拆详情区结构：Tab 化当前用例信息，同时固定详情头部的状态摘要和主操作
3. 然后补列表卡片的短 `case id`、最近运行时间、就近操作，以及轻量搜索/过滤
4. 最后完善空态 / 失败态 CTA，并重点验证缩放场景、长列表场景与新用户主流程
