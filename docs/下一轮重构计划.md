# 下一轮重构计划（Round 2）

日期：2026-03-25

> 适用范围：在你当前仓库已经基本完成 `docs/重构目标架构与里程碑.md` 的 M0–M4 之后，用于进入下一轮“把复杂度从一个超级文件继续拆开”的重构。

## 0. 本轮重构的结论先说

你现在的工程已经从“入口过载 / 单文件堆功能”回到了“可演进”的状态：`lib/app-server.js` 变成组装器，`server/` 已分层，`storage/` 已拆 repo+migration，`public/shared` 与 `public/chat` 也已出现。

**下一轮最值得投入的地方**是：把 `server/domain/conversation/turn-orchestrator.js` 继续拆成可理解、可测试、可替换的模块，让“编排、提示构造、运行时交互、持久化、事件广播”不再纠缠在一个文件里。

## 1. 目标 / 非目标 / 约束

### 1.1 业务目标

- 不影响现有本地 Chat UI 的使用体验（创建会话、聊天、Stop、Agent/Skill 编辑、卧底房间）。
- 不做破坏性 SQLite 迁移：旧数据可继续读取，升级路径稳定。
- API 路径尽量不变（前后端不需要一起大改）。

### 1.2 技术目标（本轮）

- 把 `server/domain/conversation/turn-orchestrator.js` 从“超大文件”降维为“薄编排器 + 多个职责单一模块”。
- 让 Turn 编排逻辑可以被单测覆盖：不依赖启动整站、不依赖真实 `pi`。
- 让后续新增“房间类型 / 编排策略 / tool bridge 行为”时，有明确落点（新增模块，而不是继续扩展 orchestrator）。

### 1.3 非目标

- 不强制更换技术栈（仍然 CommonJS、原生前端、SQLite）。
- 不在本轮优先改 UI 样式与所有文案。
- 不在本轮重命名所有概念（先拆边界，再做命名洁癖）。

### 1.4 强约束（延续 `docs/refactor-rules.md`）

- 不把任何新业务逻辑塞回 `lib/app-server.js`。
- Controller 只做收参/出参，不在 controller 内写复杂流程。
- 公共前端能力继续收敛到 `public/shared/`，不允许跨页面复制。
- 每个 PR 至少保留一条自动化验证路径（单测/集成/烟测皆可）。

## 2. 现状与痛点（只列本轮相关）

### 2.1 主要瓶颈

- `server/domain/conversation/turn-orchestrator.js` 仍然承载了过多职责：Turn 状态机、Agent sandbox、prompt 构造、并发/串行编排、Stop、tool bridge 交互、事件广播、session 导出、安全校验……
- 单测层面已经有 `tests/runtime/turn-orchestrator.test.js`，但目前仍偏“黑盒”，拆分时容易靠肉眼回归。

### 2.2 次要瓶颈（可选项）

- `public/app.js` 仍然偏大（虽然已有 `public/chat/*`），后续可再做一次“状态与渲染分离”。
- `npm test` 在某些受限环境可能出现 `node --test` 的进程创建限制（CI/沙盒差异），需要明确“支持的测试运行方式”。

## 3. 本轮目标架构（增量版）

本轮只要求把 Turn Orchestrator 拆到更清晰的边界里，不要求一次性达到理想树形。

建议落地结构（示意）：

```text
server/domain/conversation/
  turn-orchestrator.js              # 对外 facade，尽量薄
  turn/
    turn-state.js                   # TurnState/Stage 的结构与纯函数
    turn-runtime-payload.js         # buildRuntimePayload（纯拼装）
    turn-events.js                  # emitTurnProgress / broadcast helper
    turn-stop.js                    # stop 请求与 handle cancel 逻辑
    session-export.js               # resolveAssistantMessageSessionPath + 安全校验
    agent-sandbox.js                # sandbox dir / private dir / ensure
    agent-prompt.js                 # prompt 构造（含 mention sanitize）
    agent-executor.js               # 执行单个 agent（run/store/tool bridge 交互）
    routing-executor.js             # 并行/串行批次调度（不关心 prompt 细节）
```

> 说明：你也可以不建 `turn/` 目录，先把模块拆成同级文件；但建议尽早收拢到 `turn/` 以避免目录继续膨胀。

## 4. 拆分策略（按 PR 交付）

这部分是你可以直接照着做的“执行脚本”。每个 PR 都要求：行为不变、可回滚、可验证。

### PR-1：补“角色化回归”用例（Characterization Tests）

目的：先把最关键行为固定住，后面怎么挪文件都不怕。

范围（建议至少覆盖）：

- Stop：请求停止后，队列中未执行的 agent 不再启动；正在运行的 handle 会被 cancel。
- mention 路由：`@Agent` 在新行/尾部 mention block 的解析规则不变；串行/并行模式不变。
- session export：只能导出允许目录内的 session，路径越界要失败。
- 卧底房间：对话限制与回合推进不回归（至少跑一个 smoke）。

交付物：

- 在 `tests/runtime/turn-orchestrator.test.js` 增加 3–6 个“只关心输入输出”的断言测试。
- 必要时补最小 mock（store / skillRegistry / agentToolBridge），禁止引入真实网络与真实 `pi`。

验收标准：

- `npm run check` 通过。
- 测试新增后可稳定跑通（在你的本机/CI 环境）。

### PR-2：抽 TurnState 与纯函数（先从“无副作用”开始）

目的：把“状态结构 + 纯函数”从 orchestrator 里剥离出来，让后面拆执行逻辑更安全。

建议抽取：

- `resetTurnStage`
- `summarizeTurnState`
- `getTurnStage`
- 任何只读/纯计算函数（clipText、sanitizeReason 也可一起带走）

交付物：

- 新增 `server/domain/conversation/turn/turn-state.js`，导出上述纯函数。
- `turn-orchestrator.js` 改为引用新模块（不改行为）。
- 为纯函数补 3–5 个小单测（可放在现有 runtime 测试文件里，或新增 `tests/runtime/turn-state.test.js`）。

验收标准：

- PR 前后关键测试用例完全一致通过。
- `server/domain/conversation/turn-orchestrator.js` 行数明显下降。

### PR-3：抽“事件与运行时负载”拼装

目的：把“广播什么事件/如何拼 runtime payload”与“如何执行 turn”分离。

建议抽取：

- `emitTurnProgress`
- `broadcastConversationSummary` 的包装
- `buildRuntimePayload`
- 与 SSE 事件名称/负载结构相关的拼装

交付物：

- `server/domain/conversation/turn/turn-events.js`
- `server/domain/conversation/turn/turn-runtime-payload.js`

验收标准：

- UI 的 turn 进度更新仍正常（至少跑一次手动 smoke：聊天并观察进度/Stop）。

### PR-4：抽“session export + sandbox”相关文件系统能力

目的：把文件系统安全边界与目录规则独立出来（这类逻辑最适合单测，且非常稳定）。

建议抽取：

- `resolveAssistantMessageSessionPath`（含目录越界校验）
- `ensureAgentSandbox` / `resolveAgentSandboxDir` / `resolveAgentPrivateDir` 等
- `toPortableShellPath` 之类与 OS 路径相关的小工具

交付物：

- `server/domain/conversation/turn/session-export.js`
- `server/domain/conversation/turn/agent-sandbox.js`
- 对路径越界与目录创建补单测（只测纯逻辑，不测真实权限）。

验收标准：

- session export 功能仍可用，且不能导出越界路径（强制保留测试）。

### PR-5：拆“执行单 Agent”与“批次调度”

目的：把 orchestrator 的核心职责拆成两个模块：

1) 如何执行一个 agent（prompt、tool bridge、run/task 持久化、handle cancel、结果落库）  
2) 如何调度多个 agent（并行/串行/handoff/最大并发限制/失败策略）

交付物（建议拆法）：

- `server/domain/conversation/turn/agent-executor.js`：从 `executeConversationAgent` 开始拆
- `server/domain/conversation/turn/routing-executor.js`：把“批次/队列/并发控制”搬出去
- `server/domain/conversation/turn/agent-prompt.js`：把 prompt 构造与 mention sanitize 搬出去
- `turn-orchestrator.js` 变成“胶水层”：组装依赖 + 暴露少量 public API（`runConversationTurn`、`requestStopConversationTurn`、`buildRuntimePayload`、`resolveAssistantMessageSessionPath`、`clearConversationState` 等）

验收标准（最终验收）：

- `npm start` 可用，UI 主路径不回归：新建会话、发送消息、并行 mention、Stop、导出 session、卧底房间。
- SQLite 数据可继续读取（新库初始化与旧库升级都正常）。
- 自动化测试能覆盖：mention 路由 + Stop + session export 至少一条回归路径。
- `server/domain/conversation/turn-orchestrator.js` 从“实现全集”降为“薄 facade”（目标：不再是仓库里最大的文件之一）。

## 5. 本轮发布前检查清单（DoD）

- `lib/app-server.js` 仍然只做组装与启停；没有回填业务逻辑。
- `server/api/*` controller 不出现新增的大段流程与状态机代码。
- 新增模块都被真实引用（禁止“建空目录”式重构）。
- `npm run check` 通过。
- `npm test` 在你目标运行环境通过（若 CI 与本地不同，文档化差异）。
- 手动 smoke：聊天 + 并行 mention + Stop + 卧底房间 + session export 至少各跑一次。

## 6. 风险与控制策略（本轮）

- 风险：拆分过程中行为悄悄改变（尤其是 stop、并行批次、handoff 队列）。  
  控制：PR-1 先补 characterization tests；每个 PR 都跑同一组回归用例。

- 风险：拆分后出现循环依赖或隐式共享状态。  
  控制：把“纯函数”和“副作用（FS/DB/bridge）”分层；副作用集中在 executor 模块，其他模块尽量纯。

- 风险：重构到一半又继续堆功能。  
  控制：严格遵守 `docs/refactor-rules.md`，新需求先加到新模块边界里，再回补 orchestrator。
