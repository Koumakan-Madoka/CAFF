# `caff` 重构目标架构与里程碑

日期：2026-03-24

## 1. 结论先说

这个项目现在的主要问题不是“代码写得丑”，而是**职责边界已经失控**：

- 服务端入口同时承担 HTTP 路由、SSE、会话编排、Agent 调度、工具桥接、游戏流程、静态资源服务。
- 前端单文件同时承担状态管理、页面渲染、事件绑定、提及解析、配置面板、游戏 UI。
- 存储层同时承担 schema 初始化、迁移、校验、序列化、领域拼装。
- 新功能基本都在原文件上继续堆，已经出现重复函数、旧实现残留、新旧版本并存的迹象。

所以这次不建议“大重写”，而应该做成一轮**分阶段收口的重构**：

1. 先止血，建立边界和最小测试护栏。
2. 再拆后端，让“协议层 / 编排层 / 领域层 / 存储层”分开。
3. 再拆前端，让“页面入口 / 共享能力 / 领域组件”分开。
4. 最后补测试、迁移规范和发布约束。

目标不是把项目改成“最优雅”，而是把它改成**还能持续加功能、出了问题能定位、多人改不容易互相踩炸**。

## 2. 当前现状诊断

### 2.1 结构层面的症状

从仓库现状看，几个核心文件已经明显超出单模块应承担的复杂度：

| 文件 | 现状 | 主要问题 |
| --- | --- | --- |
| `app-server.js` | 4398 行，135 个函数 | 服务端总入口过载，协议层、业务层、运行时编排层混在一起 |
| `public/app.js` | 3591 行，124 个函数 | 聊天页几乎成了前端单体应用，状态、视图、交互、解析全部耦合 |
| `chat-app-store.js` | 1393 行 | 一个类同时做 schema、迁移、SQL、DTO 归一化、业务约束 |
| `pi-runtime.js` | 914 行 | 运行时逻辑较重，但边界相对还能继续抽象 |
| `public/styles.css` | 1355 行 | UI 样式规模已接近需要分层管理 |

### 2.2 已经出现的具体“屎山信号”

1. `app-server.js` 同时包含：
   - 请求解析与响应工具
   - `/api/*` 路由分发
   - SSE 推送
   - 会话 turn 编排
   - Agent 执行与 handoff
   - who-is-undercover 游戏流程
   - 静态文件服务

2. `public/app.js` 里已经出现**重复实现和旧逻辑残留**：
   - `profileChoiceTitle`、`profileChoiceMeta`、`setConversationProfileSelectorValue`、`setConversationProfileSelectorDisabled`、`createConversationProfileSelector` 出现了两套定义。
   - `legacyRenderConversationSettings`、`legacyRenderConversationPersonaSettings`、`legacyRenderCompactConversationPersonaSettings` 与新的渲染函数长期并存，说明重构没有收尾。

3. 前端页面脚本已经开始复制粘贴：
   - `public/app.js` 和 `public/personas.js` 都实现了 `fetchJson`、模型下拉、头像处理、文件读取等共享逻辑。
   - 这意味着未来任何一个细节修复都要改两到三份。

4. 测试护栏几乎没有建立：
   - `package.json` 里的 `npm test` 目前只是 `node --check` 级别的语法检查。
   - 对话编排、mention 路由、停止 turn、卧底游戏流程、skill 文件管理这些高风险逻辑都缺少自动回归保护。

5. 领域边界混淆：
   - “标准聊天”
   - “多 Agent 会话编排”
   - “Agent tool bridge”
   - “Skill 管理”
   - “谁是卧底”
   - “A2A / 任务持久化”

   这些能力都已经存在，但没有被收回到各自模块，导致新增一个功能时，经常会横跨 3 到 5 个大文件。

## 3. 这次重构的目标

### 3.1 业务目标

- 保持当前本地聊天 UI 可继续使用。
- 保持当前 SQLite 数据可继续读取，不做破坏性迁移。
- 保持已有 API 路径尽量稳定，减少前后端同时大改。
- 为后续继续扩展多 Agent 编排、技能管理、游戏模式留出边界。

### 3.2 技术目标

- 让单文件不再同时承担 3 种以上职责。
- 让核心流程能被单独测试，而不是只能启动整站手点。
- 让“新功能加在哪里”变得明确。
- 让未来替换 UI、替换运行时或新增房间类型时，不需要再改动总入口。

### 3.3 非目标

- 这轮不追求一次性换技术栈。
- 这轮不强制把原生前端改成 React/Vue。
- 这轮不优先解决所有命名、样式、注释问题。
- 这轮不建议先把数据库从 SQLite 换掉。

## 4. 目标架构

建议把代码逐步收敛到下面这组边界。

### 4.1 服务端目标分层

### 4.3 关键边界原则

1. HTTP 路由只负责收参与出参，不直接写业务细节。
2. 业务服务只表达流程，不直接拼 HTTP 响应。
3. Repository 只负责数据读写，不承载复杂业务判断。
4. 页面入口只负责组装，公共 UI/工具逻辑不能继续复制到多个页面文件。
5. 游戏模式要作为独立领域插件，而不是塞回聊天主流程。

## 5. 推荐里程碑

下面的阶段是按“风险最小、收益最大”的顺序排的。

### 里程碑 M0：止血与护栏

目标：先让项目进入“能重构”的状态，而不是边拆边炸。

改动范围：

- 新增重构设计文档和模块边界说明。
- 从 `app-server.js` 中先抽出纯工具模块：
  - `config`
  - `http error`
  - `request body parser`
  - `response helpers`
  - `static file serving`
- 从 `public/app.js` 中抽出共享工具：
  - `fetchJson`
  - 模型选项处理
  - 头像渲染/读取
  - toast
- 明确删除“已废弃但仍保留”的 legacy 代码策略。
- 补第一批最小回归测试。

交付物：

- 新目录骨架建立完成。
- 不改变现有行为的前提下，抽出第一批公共模块。
- 一份 `refactor-rules` 或 `contributing` 级别的边界约束文档。
- 一组最小 smoke tests。

验收标准：

- `npm start` 仍可跑通。
- 聊天、创建会话、编辑 Agent、编辑 Skill 的主路径不回归。
- 新增模块被真实引用，不是“建空目录”。
- `app-server.js` 和 `public/app.js` 至少各减少一批明显无争议的工具代码。

建议周期：1 到 2 天

### 里程碑 M1：后端拆骨架

目标：把 `app-server.js` 从“上帝文件”拆成可维护的服务端结构。

改动范围：

- 拆出独立 router 与 controller。
- 把 `handleApiRequest` 的大 if-else 路由分发改成按资源组织。
- 把 `runConversationTurn`、`executeConversationAgent` 附近逻辑抽成 `turn-orchestrator`。
- 把 SSE 广播能力收进单独 `sse-bus` 或 `event-bus`。
- 把 agent tool bridge 的上下文管理单独成模块。
- 把 who-is-undercover 的 HTTP 入口与主聊天入口解耦。

交付物：

- `server/http`
- `server/api`
- `server/domain/conversation`
- `server/domain/runtime`

验收标准：

- API 路径对前端保持兼容。
- 标准聊天 turn、并行 mention、停止 turn 仍然可用。
- `app-server.js` 仅保留 server 组装、依赖注入、启动与关闭逻辑。

建议周期：3 到 5 天

### 里程碑 M2：存储层与领域模型收口

目标：把“SQL + DTO + 业务校验 + 迁移”这团东西拆开。

改动范围：

- 将 `chat-app-store.js` 拆成：
  - schema/migrations
  - repositories
  - normalization/validation
  - service facade
- 明确 `chat_*` 和 `a2a_* / runs` 的边界。
- 让会话、消息、私信、会话参与者、任务记录拥有各自 repository。
- 为关键写操作补事务边界。

交付物：

- `storage/sqlite/connection.js`
- `storage/sqlite/migrations.js`
- `storage/chat/*.repository.js`
- `storage/run/*.repository.js`

验收标准：

- 建库、升级老库、读取历史数据都可工作。
- Repository 层不再依赖 HTTP 结构。
- 新增一个字段时，只需要改 migration + repository + DTO，不需要翻整个 store 类。

建议周期：2 到 4 天

### 里程碑 M3：前端拆页与去重

目标：把前端从“大脚本拼页面”收成“按页面和共享能力组织”。

改动范围：

- 把 `public/app.js` 拆成聊天页主入口、共享工具、子面板渲染器。
- 把 `public/personas.js`、`public/skills.js` 共用逻辑抽到 `public/shared`。
- 清理重复函数和 legacy 渲染实现。
- 将聊天页中的：
  - 会话列表
  - 消息时间线
  - 参与者面板
  - 会话设置
  - 卧底控制面板
  - mention 菜单
  
  分成独立模块。

交付物：

- `public/shared/*`
- `public/chat/*`
- 页面脚本按职责拆分

验收标准：

- 页面行为与当前保持一致。
- 同一类共享逻辑只保留一份实现。
- 删除 legacy 版本后，页面仍可运行。
- `public/app.js` 不再承担所有渲染逻辑。

建议周期：3 到 5 天

### 里程碑 M4：测试、迁移、发布硬化

目标：避免重构结束后又重新堆回去。

改动范围：

- 建立测试分层：
  - 单元测试：mention 解析、turn 输入归一化、技能路径校验、卧底胜负判定
  - 集成测试：核心 API
  - smoke test：启动服务后跑主路径
- 为 SQLite 迁移增加备份/回滚约定。
- 给关键模块加日志边界和错误分类。
- 补开发约束：
  - 新功能不能直接塞回总入口
  - 新页面不能复制共享工具
  - 领域新增能力要先放入对应模块

交付物：

- 测试目录与测试脚本
- 迁移约束文档
- 基础日志/错误分级

验收标准：

- `npm test` 不再只是语法检查。
- 核心用户路径至少有 smoke coverage。
- 改动 mention 路由、对话 turn、卧底流程时，能通过自动化测试发现回归。

建议周期：2 到 4 天

## 6. 里程碑之间的依赖关系

```text
M0 止血与护栏
  -> M1 后端拆骨架
     -> M2 存储层收口
        -> M3 前端拆页与去重
           -> M4 测试与发布硬化
```

说明：

- M1 和 M2 可以部分交叉，但建议先把服务端流程边界拆出来，再深入拆 store。
- M3 最好在 M1 稳住 API 之后进行，否则前后端会一起抖。
- M4 不能拖到非常后面才开始，M0 起码就要先补 smoke test。

## 8. 风险与控制策略

### 风险 1：边拆边回归

控制策略：

- API 先保持兼容。
- 每个里程碑都做可运行交付，不做长时间分支大爆炸。

### 风险 2：SQLite 结构改乱

控制策略：

- 先拆 repository 和 migration 管理，再改字段。
- 每次迁移都保留向后兼容路径。

### 风险 3：前端拆分时行为悄悄变化

控制策略：

- 先抽纯工具和纯渲染块。
- 聊天页优先做 smoke 回归。

### 风险 4：重构到一半又继续叠功能

控制策略：

- 在 M0 就定规则：新功能必须放到新边界里，不允许继续往总文件里塞。

## 9. 建议的执行原则

1. 采用“绞杀者模式”渐进迁移，不做一次性重写。
2. 先抽无状态工具，再抽流程，再抽存储。
3. 每个阶段结束后都保证产品可运行。
4. 先保接口稳定，再慢慢替换内部实现。
5. 每次减少一个“上帝文件”的权力，而不是试图一次解决全部问题。

## 10. 最后的判断

这个项目目前已经不是“小整理一下”能解决的阶段了，但也还没到必须推倒重来的程度。

好消息是，你已经有几块值得保留的核心资产：

- `pi-runtime.js` / `minimal-pi.js` 这条运行时链路
- SQLite 持久化基础
- 本地多 Agent 对话产品形态
- skill / persona / game mode 这些已经验证过价值的功能

真正需要重构的是**组织方式**，不是把所有已有能力判死刑。

如果按上面 4 个里程碑走，这个仓库可以从“继续堆功能一定越来越乱”，回到“能稳定演进”的状态。
